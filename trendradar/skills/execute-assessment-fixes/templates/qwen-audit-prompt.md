## 维度（自由扫描，不限以下）

以下只是起点——你认为该看的，尽管看。每个维度举了 2-3 个追问方向，但不要求逐条回答，只是帮你打开思路。

### 工程完善度
- 脚本越权：一个脚本做了本该另一个脚本做的事？模块间职责重叠？
- 胶水代码质量：临时补丁（try/except pass、硬编码回退）是否掩盖了根因？
- 抽象层次：是否在正确的抽象层做事？低层细节泄漏到高层？
- 单点故障：一个组件挂了整个链路停摆？有没有降级路径？

### 文档管理
- 过期文档：references/ 里有没有描述已删除/已重命名的功能？
- 文档冗余：同一件事多份文档各说一半？该合并的没合并？
- 文档盲区：关键决策/设计取舍没有记录，后人只能靠读代码猜？
- 引用断裂：SKILL.md 引用的文件路径已失效或指向空文件？

### 依赖与配置
- 隐式依赖：脚本依赖某个文件/环境变量但没有检查是否存在就用了？
- 配置散落：同一类配置（路径、阈值、超时）分散在 settings.py / 各脚本 / .env / cron prompt 中？
- 缺省值合理性：默认值在典型场景下是否实际可用？还是必须手动改才能跑？

### 可观测性
- 出问题能定位吗：关键环节失败时日志/退出码/产物文件能否指向根因？
- 静默失败：有没有"成功了但产出是空的""退出了但没人知道"的路径？
- 运行时可见性：管线跑到哪一步了、卡了多久——外部能不能感知？

### 版本一致性
- 声明的版本 vs 实际行为：SKILL.md 说版本 2.8.0，代码导出的 `__version__` 是否一致？
- 多副本漂移：同一份 references 在 skills/references/ 和主 references/ 各有一份，内容不同步？
- 工具链版本绑定：脚本依赖的 CLI 工具（curl/hermes/git）有沒有版本假设？

### 工具适配
- 用对工具了吗：shell subprocess 调用 Python 脚本？Python 里嵌入大量 shell 命令？
- 第三方库滥用：一个简单需求引入重量级依赖？或者反过来，自己造轮子而标准库/生态已有？
- 平台耦合：代码假设 Linux/WSL/macOS？有没有 Windows 路径硬编码？

### 代码味道
- 重复模式：同样的 3-5 行逻辑出现在 3+ 个文件中？
- 函数膨胀：超过 80 行的函数是否还承担单一职责？
- 注释债务：`# TODO`、`# FIXME`、`# 待实现` 标记了却没跟进的？
- 死代码：定义了但无调用方的函数/类/常量？

### 你的直觉
- 任何你觉得不对劲、不够干净、埋了坑的地方——不受以上维度限制
