# TrendRadar 代码审计与重构作业

你是一个工程团队负责人。请基于以下系统背景，并行组织 6 个子 Agent：

## 系统背景

部署在 WSL2 上的自动化新闻简报系统，Hermes Agent 驱动多点 cron 调度。核心 pipeline：异步 RSS 抓取→关键词分类→AI 翻译→Markdown 渲染→字节分片投递企业微信。双代码副本（`~/TrendRadar/` 工作副本 + `~/.hermes/trendradar/` cron 运行时副本）。Python 3.14t，外网走 HTTP 代理。

## 3 个审计子 Agent

| Agent | 审计重点 |
|-------|---------|
| **架构审计** | 模块边界、数据流方向、单点故障、双副本设计的必要性、pipeline 各阶段耦合度 |
| **安全&配置审计** | API 凭证加载路径、git 跟踪内容的敏感性、跨环境配置漂移、shell 注入面 |
| **代码健康审计** | 死代码、`except: pass` 异常吞噬点、可观测性缺口、测试套件中长期挂起失败、import 死锁 |

## 3 个执行子 Agent

| Agent | 职责 |
|-------|------|
| **重构师** | 代码和文档的结构化、可扩展化、去耦合化、精简化。拆大模块、统一接口风格、消除重复逻辑 |
| **Performance** | 性能优化。pipeline 各阶段耗时 profiling、翻译批处理耗时曲线、网络超时参数调优、DB 写入模式优化 |
| **文档精简** | 重构 Skill 文档和 references。原则：尽量多引用（指向已有文档），少存档（删除冗余副本），引用文档本身从简，篇幅不要太长 |

## 输出要求

审计 Agent 按 🔴 严重 / 🟡 警告 / 🔵 观察 评级独立出报告。执行 Agent 产出具体 PR 级别的修改建议。你最后汇总，标记跨 Agent 的交叉发现。
