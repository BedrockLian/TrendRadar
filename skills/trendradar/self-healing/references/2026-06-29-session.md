# 2026-06-29 自动体检实战：Windows 兼容性 + 看门狗时序竞态

## 背景

用户进入 session 后要求运行自动体检。发现 Gateway 不在跑，启动后还有 4 个"cron job 未注册"警告。

## 发现问题

### 1. check_gateway() 在 Windows 上永远报假阳性

health_check.py 的 `check_gateway()` 只支持 Linux systemd 检测 + Unix socket 检查。Windows 上两条路径都失败 → 永远报 `hermes-gateway 未运行`。

**修复**：加 `if sys.platform == 'win32'` 分支，用 `subprocess.run([sys.executable, '-m', 'hermes_cli.main', 'gateway', 'status'])` 检测。

**影响范围**：live 脚本 + skill 模板 `templates/trendradar_health_check.py` 都需更新。

### 2. CRON_JOB_NAMES 与实际 job 名称漂移

health_check.py 硬编码的 job 名称与 `hermes cron list` 输出不匹配：

| 检查列表中的 name | 实际 Name: 行 | 是否命中 |
|---|---|---|
| `推送降级看门狗` | `TrendRadar 推送看门狗` | ❌（多了"降级"） |
| `月度趋势报告` | `TrendRadar 月度报告` | ❌（"趋势报告" vs "报告"） |
| `slot_direct_push` | 不存在 | ❌（脚本已删除） |

**子串匹配**：`'推送降级看门狗' in 'TrendRadar 推送看门狗'` → False。
**退化 token 匹配**：`'推送降级看门狗'.split()` 只有一个 token `['推送降级看门狗']` → 还是 False。

**修复**：删除 `slot_direct_push`，`推送降级看门狗`→`推送看门狗`，`月度趋势报告`→`月度报告`。

**教训**：CRON_JOB_NAMES 必须与 hermes cron list 实际输出保持同步。每次 cron 重建后手动检查。

### 3. 看门狗时序竞态 — delivery_watchdog 跑在管线前面

**时序**：
- 12:03:28 — 推送看门狗运行 → 查 `archive/2026-06-29/noon.md` → 不存在（管线还没跑完）
- 12:03:51 — LLM 日报推送生成 `noon.md`（30 条，9373 字节）
- 最终：archive 存在，marker 不存在，用户没收到午报

**根因**：`delivery_watchdog`（no_agent, `0 9,12,21`）和 LLM pipeline（LLM, `0 9,12,21`）共享同一 cron 调度。no_agent 脚本启动快，LLM agent 需要时间 fetch/translate/render。看门狗跑完时管线刚启动。
